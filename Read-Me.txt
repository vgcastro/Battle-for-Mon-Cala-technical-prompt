This project is broken up into two types of files, .py and .html files. 
dbBuilder.py builds the database. This python files builds an SQLite3 database
wihin the same directory in which the file sits. It then takes aws an input 
.tsv files from IMDB.com and loads the database according to the schema described
in the file. The imdbAPI.oy file actually builds the API. This file uses
Flask as a framework to build a simple web-server which is connected to the 
database built in dbBuilder.py. It is a REST API that focuses on Serving the 
user data from the database and allowing the user to update the database (according
to the prompt). This web-server is bare bones. It is suffucient enough to allow 
interaction with a user in order to test the functionality. We use some very basic templates 
to serve data and to retrieve data from users. There are API testers out 
there (such as Postman) but I focused on functional checks. For the sake of time, the
API does not conduct very much input validation or error handling, but it can be added 
at a later point and the framework provides that ability. Additionally, it does not 
build security into it with the assumption being that this is a development server and
if it were to be placed into production, implementation of AuthO or some other security
framework would be implemented. Additionally, a production server would use a better database 
such as PostgreSQL along with a better server such as NGINX running on Gunicorn.
Use the code by running dbBuilder.py and afterwards running imdbAPI.py. imdbAPI.py will 
start the server and allow you to interact with it via local host and the endpoints specified in the code according 
to their functions. Below I list some functional tests demonstrating the functionality of the code and ability to 
complete the required tasks. Additionally, anyone who desires to run this code must download and place
into the same directory (to run the code as is) the following files from imdb: title.basics.tsv  name.basics.tsv

Limitations:
-Based on the schema, this program did not take a year a movie was made into consideration. Consequently, even though every movie contains a unique movie identifier, some of the names may match (ie. Jumanji). If we were conducting queries soley using
MovieID, it wouldn;t be an issue, but since users query movies by names, sometimes the query may result in two unique identifiers for what seems to be one movie but in reality are two or more.
